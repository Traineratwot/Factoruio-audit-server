# Plan: Queue pending mods for immediate fetchFullInfo

## Problem

`SyncModsJob` / `sync:mods` runs daily and updates `latest_version` in the `mods` table. But `FetchFullInfoJob` (runs every 5 min, limit 10) uses a 7-day cooldown per mod and picks by popularity — so newly added mods or mods with a version bump wait days before getting full info.

## Approach

Add a `pending_full_info` boolean column to `mods`. When `sync:mods` detects a new mod or a `latest_version` change, mark it. Modify `FetchFullInfoJob` to process pending mods first (no cooldown).

No separate queue table needed — the existing `mods` table + `FetchFullInfoJob` already runs every 5 min with `limit: 10`.

### 1. Migration: add `pending_full_info` to mods table

**File**: `database/migrations/2026_07_05_000000_add_pending_full_info_to_mods_table.php`

- Add boolean column `pending_full_info`, default `false`, indexed

### 2. Modify `ModsCommand::handle()`

**File**: `app/Console/Commands/Sync/ModsCommand.php`

Before each `Mod::upsert()` batch:
- Query existing mods' `latest_version` for names in the batch
- After upsert, compare old vs new `latest_version` for each mod
- If `latest_version` changed (or mod was new), set `pending_full_info = true`

Implementation:
```php
// Before upsert: get current versions
$existingVersions = Mod::whereIn('name', $batch->pluck('name'))
    ->pluck('latest_version', 'name');

// After upsert: detect changes
$modNames = $chunk->pluck('name');
$updatedMods = Mod::whereIn('name', $modNames)
    ->where('pending_full_info', false)
    ->get();

$pendingIds = [];
foreach ($updatedMods as $mod) {
    $newVersion = $chunk->firstWhere('name', $mod->name)['latest_release']['version'] ?? null;
    if ($newVersion && $newVersion !== $mod->latest_version) {
        $pendingIds[] = $mod->id;
    }
}
// Also add newly created mods (not in existingVersions)
// Then: Mod::whereIn('id', $pendingIds)->update(['pending_full_info' => true]);
```

Also mark brand-new mods (not in `existingVersions`) as pending.

### 3. Modify `FetchFullInfoJob::handle()`

**File**: `app/Jobs/FetchFullInfoJob.php`

Before the existing popularity-based query, add a priority pass:
- Query mods where `pending_full_info = true`, ordered by `popularity DESC`
- Process these first (respecting `maxRequests` and `delayMs`)
- After processing, set `pending_full_info = false`
- If any budget remains, fall through to the existing cooldown-based logic

### 4. Same change in `ModsCommand` + `SyncModsJob`

`SyncModsJob` is a copy of `ModsCommand` logic. Apply the same pending detection to both.

**File**: `app/Jobs/SyncModsJob.php`
